 [Feature] 增加本地断点续传功能，完美解决 524/429 报错导致的进度丢失问题

**详细说明：**
在翻译大文本文件时，如果遇到 API 服务商极其不稳定的情况（如频繁报 524 Timeout、429 限流等），原有的重试机制耗尽后任务会被取消，导致已经成功翻译的大量宝贵记录全部丢失。为了解决这个问题，我实现了一个极简的本地断点续传系统。

### 💡 核心修改点
1. **全局翻译记忆库 (`agent.py`)**：使用纯原文作为 Key，将每次请求的结果（无论成功还是耗尽重试的失败）实时追加写入 `/app/output/translation_checkpoint.jsonl`。
2. **防误删与自动清理**：任务结束时，联动原生的 `self.unresolved_error_count`。只有当该计数器为 0（即彻底无报错）时，才会自动销毁临时文件，保持硬盘清爽。
3. **支持断点恢复**：无需额外操作，网页端原样重新发起翻译即可。系统会比对文件，秒速跳过已成功的段落，直接从失败处接力翻译。
4. **修复 Docker 构建 (`Dockerfile`)**：修改了构建逻辑，确保本地修改的代码能被正确打包进镜像。

### 🛠️ 如何测试我的修改版本？

你可以直接克隆我的 Fork 仓库进行测试。为了防止和旧版本冲突，请按照以下步骤全新部署：

**1. 彻底清理旧环境与数据（注意备份成品）**
```bash
docker stop docutranslate && docker rm docutranslate
rm -rf ~/docker_sources/docutranslate/output/*
```

**2. 拉取我的修改版仓库并强制构建**
```bash
# 克隆包含断点续传功能的 Fork 仓库
git clone https://github.com/wu1552944668/docutranslate.git
cd docutranslate

# 强制无缓存构建
docker build --no-cache -t docutranslate:latest .
```

**3. 启动并映射输出目录**
```bash
mkdir -p ~/docker_sources/docutranslate/output
chmod 777 ~/docker_sources/docutranslate/output

docker run -d \
  --name docutranslate \
  --restart=always \
  -p 8010:8010 \
  -v ~/docker_sources/docutranslate/output:/app/output \
  docutranslate:latest
```

**测试方法：**
故意设置一个极低的超时时间或速率限制来触发任务失败。任务停止后，直接在网页上再翻译一次同一个文件，观察后台日志打印的 `已恢复 XX 条匹配当前任务的记录` 即可验证。
